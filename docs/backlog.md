# バックログ（残タスク）

## 未対応

### #6 松井スクレイプの堅牢化（信頼性向上）

**状態**: 対応中（2案実装済み・CI 実測で効果確認予定 / 優先度: 中）

**問題**
`engine/fgi/fetchers/matsui.py` の Playwright スクレイプ（信用評価損益率・買い方）が、
CI 実行でしばしばタイムアウトする。エラーは概ね以下:

```
Page.wait_for_selector: Timeout 75000ms exceeded.
  - waiting for locator("tr:has-text('買い残')") to be visible
```

3回リトライ（各 75s まで延長）しても全滅する回がある一方、成功する回は正常に値と
基準日（"7/16(木)" 形式）を取得できる。**成功/失敗が実行単位でまとまる**傾向があり、
セレクタ構造の不一致ではなく **サーバ側スロットリング/ボット対策や描画遅延**が原因と推測。
（2026-07-18 の検証では短時間に多数アクセスしたため失敗頻度が上がった。日次1回運用なら
頻度は下がる見込み。）

**現状の緩和策（実装済み）**
- 確定日次履歴 `engine/fgi/data/margin_pl_history.csv`（松井・正データ）を最優先で使用。
  スクレイプが失敗しても #6 の表示・スコアは正しい。
- スクレイプ成功時は基準日でキーし、失敗/基準日不明なら捏造せず見送り（自己修復）。
- `page.goto` の `wait_until` を `domcontentloaded` → **`load`**（全リソース読込完了）に変更。
- 買い残行を待つ前に、より早く安定描画される見出し（`信用取引指標` / `信用残速報`）の
  出現を先に待つ **段階待ち**（`_wait_for_heading`, 上限 `_HEADING_TIMEOUT_MS=20000`）を導入。
  完全ブロック/未描画時は見出し待ちで早く失敗し、満額の行待ちに入らず次リトライへ回す。

**検討する改善案（未実施）**
- リクエスト間隔を空ける／User-Agent・ヘッダを実ブラウザ寄りに調整（ボット対策回避）。
- レンダリング済み HTML を1回だけ取得できる代替（軽量プロキシ/キャッシュ）を検討。
- 失敗時に日次履歴 CSV を手動追記する運用フロー（暫定）をドキュメント化。

**検証方法**
`.github/workflows/daily.yml` を `workflow_dispatch`(real) で複数回起動し、
`fetch_errors.margin_pl_ratio_matsui` の発生率と、成功時に基準日が正しくキーされるか
（`web/public/data/series/margin_pl_ratio.csv` に基準日で追記されるか）を確認する。
※ 検証アクセスは短時間に集中させない（スロットリングを避ける）。
